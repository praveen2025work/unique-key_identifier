Developer Specification · Action / All / My History Tabs

Business use case, role-based filter rules, data contract, and acceptance criteria for the Exception Hub workflow UI.

1. Problem & Goal

Today the Exception Hub shows a single My Actions toggle. Users cannot easily see (a) items waiting on them right now, (b) the full set they have visibility over, and (c) items they already actioned that have moved downstream. The redesign replaces the single toggle with three mutually-exclusive tabs driven by the user's role and the current active workflow step on each exception.

Goal: a user at any L4 role (Analyst, Reviewer, Approver, etc.) lands on their work queue by default, can pivot to the full scope when investigating, and can trace what they have already processed — all without filters or context menus.

2. Tabs & Definitions

Tab Default? Definition
Action Required
key: action		 Yes (L4 roles) Exceptions where the currently active workflow step is mapped to the logged-in user's role. Only items needing this user's intervention. Count badge drives the red urgency signal.
All Visible
key: all		 Yes (Owner) Every exception the user has entitlement to view, regardless of current step. For L4 users this is a read-only investigative view; action button only appears on rows where they are the active actor.
My History
key: history		 No Exceptions this user has previously actioned (approved / rejected / reviewed / commented) but which are no longer in their queue — typically moved to the next step or closed. Sourced from the workflow audit trail, not from current-state.

3. Role Behaviour Matrix

Role Default Tab Action Required filter All Visible filter My History filter
PC_OWNER
Owner / Admin		 all All exceptions where any step is currently active (i.e. not terminal). Owner is not gated by step ownership. Everything in entitled scope. Exceptions the Owner user has personally actioned (rare but supported).
ANALYST, REVIEWER, APPROVER, …
any L4 functional role		 action exception.currentStep.assignedRole == user.role
AND exception.currentStep.status == 'IN_PROGRESS'		 Everything in entitled scope, action button disabled on rows where the user is not the active actor. audit.actorId == user.id AND audit.step != exception.currentStep
Entitlement scope is a separate filter applied on top of every tab (SDS book access, legal entity access, region, etc.) — it is enforced by the existing Central Enterprise Entitlement System and MUST be applied server-side before any tab filter runs.

4. Filter Logic (server-side pseudocode)

// ExceptionQueryService.java — new endpoint param: ?tab=action|all|history

public Page<ExceptionDto> list(User user, Tab tab, Pageable page) {

    // 1. Always start from the user's entitled scope.
    Specification<Exception> spec = withinEntitlement(user);

    switch (tab) {

        case ACTION:
            if (user.hasRole("PC_OWNER")) {
                spec = spec.and(hasAnyOpenStep());
            } else {
                spec = spec
                    .and(currentStepAssignedTo(user.getRole()))
                    .and(currentStepStatus(IN_PROGRESS));
            }
            break;

        case ALL:
            // No step filter; entitlement already applied.
            break;

        case HISTORY:
            spec = spec.and(hasAuditBy(user.getId()))
                       .and(currentStepNotAssignedTo(user.getRole()));
            break;
    }

    return exceptionRepository.findAll(spec, page);
}

currentStepAssignedTo joins on the workflow's workflow_step table where is_current = true. hasAuditBy joins on the workflow_audit / Envers _AUD tables by actor id.

5. API Contract

List endpoint

GET /api/exceptions
    ?tab=action|all|history          // required
    &status=OPEN,IN_PROGRESS         // optional
    &page=0&size=100&sort=aging,desc

Counts endpoint (for tab badges — cheap, cached)

GET /api/exceptions/counts
Response:
{
  "action":  47,
  "all":     2478,
  "history": 312
}

Counts endpoint must respect the same role + entitlement rules as the list endpoint. Call it on page load and re-poll (or refresh via the existing SSE/WebSocket if workflow events are already pushed) when a user submits an action.

6. Frontend Structure (React / TypeScript)

// Suggested component tree in the existing Exception Hub React app
<ExceptionHubDashboard>
  <Toolbar />
  <ActionTabs
      value={tab}                         // 'action' | 'all' | 'history'
      counts={counts}                     // from /counts endpoint
      onChange={setTab}
      userRole={user.role}
      defaultTab={user.role === 'PC_OWNER' ? 'all' : 'action'}
  />
  <ContextStrip tab={tab} role={user.role} />
  <ExceptionTable
      query={{ tab, filters, sort, page }}
      rowRenderer={row => (
        <ExceptionRow
            row={row}
            isMine={row.currentStep.assignedRole === user.role}
            canAction={row.currentStep.assignedRole === user.role
                       && row.currentStep.status === 'IN_PROGRESS'}
        />
      )}
  />
</ExceptionHubDashboard>
  • Persist tab in URL (?tab=action) so deep-links and browser back work.
  • Row highlighting: rows where isMine get the yellow left-border treatment (see mockup). This is especially valuable on the all tab so a user can spot their own items.
  • Action button is disabled with tooltip "Awaiting {roleName}" when canAction === false.
  • Empty states: each tab needs its own copy (see mockup — swap through the tabs when the count is 0).

7. Acceptance Criteria

  1. Owner user lands on All Visible by default; sees global counts on all three tab badges.
  2. Any L4 role user lands on Action Required by default; badge equals count of exceptions where current step is assigned to their role AND status is IN_PROGRESS.
  3. Switching to All Visible shows the user's full entitled scope; rows not awaiting them render the action button as disabled with the correct tooltip.
  4. Switching to My History shows only exceptions the user has actioned AND which have since moved past their step. Same exception never appears in both Action Required and My History at the same time.
  5. After a user submits an action, the row disappears from Action Required and appears in My History on the next refresh; both badge counts update.
  6. Tab state is preserved in URL; refreshing the page preserves the current tab.
  7. Entitlement denial (e.g. user loses SDS book access mid-session) must drop the row from all three tabs on next load.
  8. Audit: every list request logs {userId, role, tab, resultCount} to the existing audit sink.

8. Edge Cases to Confirm with BA

  • Multi-role users — if a user holds both Reviewer and Approver, is Action Required the union of both queues? (Recommended: yes, with the step column disambiguating.)
  • Rework / re-open — if an approver rejects an item back to the analyst, does it reappear in the analyst's Action Required and also stay in the approver's History? (Recommended: yes — History is "I have touched this", not "I have finally cleared this".)
  • Delegation — during leave coverage, should the delegate's Action Required include the delegator's queue? Needs product decision; not in scope for v1.
  • Closed / terminal exceptions — should they appear in History indefinitely or be capped (e.g. 90 days, with archive beyond)? Recommended: 90 days visible, archive accessible via existing search.
