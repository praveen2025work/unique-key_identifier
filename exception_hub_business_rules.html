Business Document EH-RULES-004 · DRAFT

Exception Hub
access, views & notifications

A working reference for business and technology teams. Covers how the application identifies who you are, what you see in each of the three dashboard views (Actions Required, All Exceptions, My History), how notifications fan out per L4, and the parallel Admin role for managing notification and SLA configuration.

S 01 · How the app finds your role

Roles live in finportal. Exception Hub looks them up.

The user logs in to finportal first. When they open Exception Hub, the application checks for the user's TB BB roles. Roles are not stored anywhere inside Exception Hub long term. They are sourced fresh from finportal so that any role change made in finportal is reflected the next time the user logs in.

The lookup sequence

01
UI calls the local entitlement serviceReturns the user's TB BB roles if they are already on file in our database.
02
If no roles found, the Python filewatcher service is asked to fetch fresh entitlementsThe filewatcher calls the finportal entitlement API for the user's BRID, persists any returned roles into our database, and returns them to the UI.
03
If finportal does not know the user, return 403 ForbiddenThis is a hard stop. The user has no business identity in finportal, so the application cannot grant access.
04
If finportal knows the user but no TB BB role is present, block at the landing screenThe user is recognized but holds no role that grants access to Exception Hub for TB BB.
05
If a TB BB role is present, save it locally and route to the dashboardThe dashboard then shows the three views described in S 02.

Why this sequence is interim. The current design pulls roles per user on demand. Once finportal can deliver the full entitlement set as a daily file (or via Autosys extract), the Python filewatcher will switch to processing that bulk feed nightly, and per user lookups will only run as a fallback for truly new users.

One user can hold up to two roles. The first is a primary role from the list (Front Office, Product Control, Risk, Regulatory, Reg Policy, IT, or Governance Forum). The second is the optional Admin role described in S 06. The Admin role grants no exception visibility. It only unlocks the configuration screens.

S 02 · The three dashboard views

Every user sees three lists. What's in each list depends on the role.

The dashboard splits exceptions into three views. The headers are the same for everyone, but the contents follow rules that vary by role.

Actions Required

Items currently waiting on this user. The work queue.

All Exceptions

Everything the user is entitled to see, regardless of who is acting on it now.

My History

Every exception this user has acted on personally.

The rule per role, side by side

Two FO users, same book. When two FO users have access to the same master book (one as FO Owner, the other as Delegate), both see all exceptions on that book in All Exceptions. Both also see any pre workflow exception in Actions Required. Whoever clicks first to start the workflow gets it in their My History. After that, the exception leaves Actions Required for both, and the other user still sees it in All Exceptions but not in their personal history.

S 03 · When can a user act

Seeing an exception is not the same as being able to move it forward

Visibility puts a row on the screen. Acting on a row, meaning approving, rejecting, sending back, or advancing the workflow, depends on three checks. All three must be true for the action buttons to enable.

Check 1The user holds a TB BB role
+
Check 2The active step is configured for that role
+
Check 3The step is the active step on this exception right now
=
Action allowedButtons enabled

Two consequences worth noting. Governance Forum users always fail Check 2, so they never act. Product Control users see exceptions at every stage including FO Review and Risk Review, but their action buttons enable only when the active step lands on a PC step.

S 04 · Notifications and SLA escalation

Two triggers, one wide notification table

The Business Area L4 Notification table holds, for each L4, one column pair per role: an Owner column with the recipient address (a BRID or a distribution list), and a Supervisor column used only on SLA breach. Layout looks like this:

business_area_l4_notification (transposed by role)

L4 pc_owner pc_supervisor credit_risk_owner credit_risk_supervisor reg_reporting_owner reg_reporting_supervisor ...
FX_OPTIONS_AMER DL_PC_FXAMER@bcs sup.fxamer.pc@bcs DL_CR_FXAMER@bcs sup.fxamer.cr@bcs DL_RR_FXAMER@bcs sup.fxamer.rr@bcs ...
RATES_EMEA DL_PC_RATESEMEA@bcs sup.ratesemea.pc@bcs DL_CR_RATESEMEA@bcs sup.ratesemea.cr@bcs DL_RR_RATESEMEA@bcs sup.ratesemea.rr@bcs ...

Real table has owner and supervisor pairs for every role: PC, the five Risk sub roles, Reg Reporting, Reg Policy, IT central, plus a single column for Governance Forum (no notifications).

Step transition (regular)

An exception advances to a new step. The role that owns that step gets notified.

To owner column for that role on this L4

Supervisor is not included on regular triggers for any role.

SLA breach (escalation)

The step has been pending longer than its configured SLA. The same recipients are notified, with the Supervisor added in cc.

To owner column for that role on this L4 Cc supervisor column for the same role on this L4

SLA thresholds are configured per workflow step in the SLA configuration table, managed by the Admin role.

The notification table is not used to filter visibility. It only decides where notification emails go. The same row can list five different distribution lists. None of those lists changes what their members see when they log in. Visibility is governed by S 02. Notifications are a separate concern.

S 05 · The Admin role

A parallel role for configuration management

ADMIN-TB BB-USER-RW

A dedicated finportal role created specifically for managing Exception Hub configuration. It runs in parallel to a primary role. For example, a user holding Product Control-TB BB-User-RW can also hold ADMIN-TB BB-USER-RW. Their primary role still governs what exceptions they see. The Admin role unlocks two configuration screens that are otherwise hidden.

What Admin can do

  • Maintain the Business Area L4 Notification table. Add or amend the owner and supervisor entries per role per L4.
  • Maintain SLA configuration. Adjust the time thresholds that determine when a step is considered breached.

What Admin does not do. The Admin role grants no additional exception visibility. There is no business area level entitlement attached to it. A PC plus Admin user still sees only PC's L4 in the exception views. Admin is global for configuration only.

Reading the dashboard preview

alice and anna both have access to MB_4471. Click alice, then click anna in the user row above. Their tab counts are identical for Actions Required and All Exceptions, but their My History counts differ. Their All Exceptions row is identical (six exceptions on MB_4471). Their Actions Required is also identical, both seeing the one pre workflow exception EX-2001. Their My History is different. alice initiated four exceptions (2002, 2003, 2008, 2009). anna initiated only one (2004). The same access set, but personal action history diverges.

paul carries the Admin badge. In Actions Required he has the three exceptions currently at PC Review (2002, 2004, 2009 with the breach). In All Exceptions he has the full six on his L4 including the pre workflow EX-2001. In My History he has the two where he completed his PC step (2003, 2008). The Admin role does not change any of these. It only unlocks the configuration menu.

rachel has a typical Risk queue. She has Actions Required on EX-2003 and EX-2010 (both at Credit Risk Review). Her All Exceptions includes those plus three more where Credit Risk has already completed (2006, 2007, 2008, all moved past Credit Risk by her own actions). My History shows only the three she actually approved. EX-2003 and EX-2010 are in her queue but not yet acted on.

rita has a tight queue. Two exceptions at Reg Reporting Review (2006, 2008). One closed exception where she acted (2007). Her All Exceptions is just those three. EX-2003 and EX-2010 are at earlier stages, so she does not see them.

gary the Governance Forum user is the bookkeeper. All ten exceptions appear in his All Exceptions tab. His Actions Required and My History are completely empty. He is read only by design.
