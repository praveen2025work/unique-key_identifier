Executive Summary

The Trading Book / Banking Book (TBBB) application is the firm's system of record for position-level exceptions across trading and banking book classifications. It ingests exceptions from the Atlas source system, coordinates enrichment with the downstream Finstore platform, and supports manual and reassignment workflows originated by Finance operators.

This document describes the exception lifecycle across three channels — systematic, manual, and reassignment — and the daily 4:00 PM EST handshake with Finstore that consolidates regional feeds (Tokyo, London, New York) before enrichment. It concludes with the UAT coverage model currently validating the integration.

Core Design Principle

Every exception — regardless of origin — must transit Finstore for enrichment before it is loaded back into TBBB. The daily file is always sent, even when no manual or reassignment records exist, to preserve Finstore's deterministic consolidation window.

02

System Context

Three applications participate in the exception lifecycle. Atlas is the upstream source for systematic position exceptions. TBBB owns classification, aging, user workflows, and the system of record. Finstore is the enrichment hub that consolidates regional feeds and returns a single enriched file.

Atlas

Position-level source system. Generates systematic exceptions that flow directly into Finstore as the global reference feed.

  • Role: upstream source of truth
  • Output: systematic exception events
  • Ownership: upstream Finance platform

TBBB (this application)

Manages trading/banking book classification, aging, manual uploads, and reassignment workflows. System of record for exception disposition.

  • Inputs: Atlas systematic, manual UI uploads
  • Outputs: daily manual + reassignment file to Finstore
  • Inputs: enriched file from Finstore (post-consolidation)

Finstore

Enrichment and consolidation engine. Waits for all regional feeds (Tokyo, London, New York) plus TBBB's global file, then enriches and returns the master file.

  • Consolidates: TKY + LDN + NYC + TBBB global
  • Integration: SFTP-based file exchange
  • Cadence: single daily enrichment window

Integration Layer

SFTP exchange for inbound/outbound files. TBBB watches a defined inbound path for the enriched Finstore return file and writes to an outbound path for the daily manual+reassignment export.

  • Transport: SFTP
  • Trigger: daily scheduler (16:00 EST outbound)
  • Watcher: TBBB FileWatcher on inbound path

End-to-End Flow

The diagram below traces the complete lifecycle: systematic exceptions flowing from Atlas through Finstore enrichment back into TBBB, alongside the daily outbound file carrying manual uploads and TB⇄BB reassignments. The 4:00 PM EST cut-off is the deterministic gate.

Flow Narrative

#StepDescription
1 Atlas → Finstore Systematic exceptions are emitted by Atlas directly to Finstore as part of the global feed. TBBB does not intermediate this path.
2 Manual → TBBB Finance operators upload manual exceptions through the TBBB manual upload screen. Records land in staging, then master tables.
3 Reassignment → TBBB TB⇄BB reclassifications are executed in TBBB. Reassigned positions have their classification tagged as REASSIGNMENT.
4 TBBB Daily Export At 16:00 EST, TBBB writes a consolidated outbound file containing manual exceptions + reassignments to Finstore SFTP — empty file if none.
5 Finstore Consolidation Finstore waits for TKY, LDN, NYC regional feeds plus the TBBB-GBL file, then enriches the combined dataset.
6 Finstore → TBBB Finstore writes the enriched master file to SFTP. TBBB's FileWatcher picks it up, loads master tables, and aging begins.
04

Exception Types & Classification

Three channels converge in TBBB's master tables. Each has a distinct origin and lifecycle but all ultimately flow through Finstore for enrichment before aging begins.

Channel Origin Path to Finstore Classification / Notes
Systematic Atlas Direct (Atlas → Finstore) Auto-generated position exceptions. Enriched by Finstore and returned to TBBB in the daily master file.
Manual Upload TBBB UI Via TBBB daily file Operator-entered exceptions not sourced from Atlas. Loaded to staging → master; carried in the 16:00 EST outbound file for enrichment.
Reassignment TBBB workflow Via TBBB daily file TB⇄BB reclassifications. TBBB updates position classification = REASSIGNMENT and includes the record in the outbound file alongside manual exceptions.
Why the daily file is always sent

Finstore's consolidation window depends on receiving the TBBB-GBL file as a deterministic arrival signal. An empty file still satisfies the handshake. Skipping a day because there are no manual or reassignment records would stall the global enrichment cycle.

UAT Coverage

UAT validates that each channel correctly flows through the enrichment handshake and that edge cases — empty files, reassignment tagging, late regional feeds — are handled deterministically. The matrix below summarises the scenario groups currently in execution.

Core Validation Scenarios

ID Scenario Validation Focus Priority
U01 Systematic exception end-to-end Atlas → Finstore → TBBB enriched load; verify field-level enrichment and master table persistence. Critical
U02 Manual upload — populated file UI upload → staging → master → 16:00 EST outbound; confirm record appears in enriched return. Critical
U03 Manual upload — empty day No manual records and no reassignments: confirm empty file is still written to Finstore SFTP at 16:00 EST. Critical
U04 Reassignment TB → BB Classification flips to REASSIGNMENT; record included in outbound file; Finstore treats as reclassification not new exception. Critical
U05 Reassignment BB → TB Mirror of U04 in opposite direction; verify idempotency if reversed same day. High
U06 Mixed day — manual + reassignment Single outbound file carries both record types; downstream parsing distinguishes them correctly. High
U07 Enriched file inbound load FileWatcher detects Finstore return; master tables updated; aging advances on day N+1. Critical
U08 Late regional feed Finstore holds enrichment until all regional files arrive; verify TBBB does not receive partial enriched data. High
U09 Malformed outbound file Simulate schema break; confirm Finstore reject is logged and operations alerted. High
U10 Duplicate manual upload Same record uploaded twice in a day; verify dedup rules at TBBB staging layer. Medium
U11 Aging continuity post-enrichment Day-N exception confirmed to age correctly on Day N+1 after enriched reload. High
U12 16:00 EST scheduler resiliency Delayed scheduler / missed cut-off handling; verify alerting and manual trigger path. Medium
