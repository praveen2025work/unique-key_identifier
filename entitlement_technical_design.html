§ 01 — Architecture

Two pathways feeding one filtered view

Two source feeds — NPL Ramp data and the Business Area Governance table — produce two entitlement tables. The main exception table is unchanged. At query time the app picks one join based on the logged-in user's role, and shows them only the exceptions they're entitled to.

Figure 1 — Sources on the left, derived entitlement tables in the middle, the app on the right. The app picks the join based on the user's role; the user info table is the single source of truth for "what role am I right now."

§ 02 — The 5 → 3 Collapse

Why one source row becomes up to four entitlement rows

The NPL Ramp feed gives us five columns. The entitlement table needs three. The collapse isn't lossy — it's a reshape. Each non-null person on a master book becomes their own entitlement row, tagged by which seat they hold (owner-side or supervisor-side).

Why split rows instead of one wide row?

Putting four GIDs across two columns on one row would force every downstream join to do four OR comparisons. Splitting into rows keeps the join trivial — "is this user's GID anywhere in the FO_Owner or Supervisor column for this book?" — and lets us add or remove people without rewriting the row shape.

What about the empty cells?

Each entitlement row carries exactly one populated GID — either FO_Owner or Supervisor, never both. The empty column makes the row's intent obvious to anyone reading the table and lets the workflow gate distinguish owner-actions from supervisor-actions cleanly.

Edge cases

Same person in two seats. If the NPL Owner and the supervisorGID are the same GID, they get two rows — one as FO_Owner, one as Supervisor. They earn access through both seats and the workflow can recognize either.

Null in the source. If jointSupGID is empty, no Row 4 is generated. There is no "ghost" entitlement.

Replay. The transformation is keyed by MasterBookId + load batch. Re-running the same batch tombstones and rewrites the same set of rows — never duplicates.

§ 04 — Roles & Login Flow

Roles come from Entitlement Service, refreshed on every login

The app does not store role definitions. It reads the user's current roles from the central Entitlement Service the moment they log in, and writes them into a local user info table for fast lookup during the rest of the session.

Figure 3 — Login flow. Roles are pulled fresh on every login; the local user info table is a session-friendly cache, never the source of truth.

Why refresh on every login

If a user changes role this morning in the central Entitlement Service, the change should take effect the next time they log into the app — not when an overnight sync runs. Pulling on login keeps the app honest without a heavy real-time integration.

Why a local user info table at all

Every page the user opens needs to know their role to pick the right join and gate the right workflow actions. A local upsert per login means we read role data once over the network and then hit a local table for the rest of the session — fast, simple, no per-page calls to Entitlement Service.

§ 05 — Role-Based View Resolution

One simple decision: which join to use

Every time the user opens the exception list, the app checks their role on the user info table and picks one of two join paths. There is no third option, no hybrid, no fallback.

Figure 5 — Role-based join selection. The role on the user info table picks the path; the path picks the rows; the rows are joined to the unchanged main exception table.

FO Owner — what they see

Every exception whose sdsbookcode matches a master book where they appear as FO_Owner or Supervisor in the FO entitlements table. No L4-level access; their world is master-book-scoped by design.

Other roles — what they see

Every exception whose businessarea matches a Business Area L4 where they're configured as the relevant role's Owner/Delegate or Supervisor in the L4 entitlements table. Master-book-level access does not apply.

Multi-role users

If a user has more than one role (e.g. Reg Policy and Product Control), the L4 join produces the union of exceptions across both roles. The user info table stores the full set; the resolution logic does not pick one and ignore the others.

§ 06 — Workflow Action Gating

Seeing an exception ≠ being able to act on it

Once an exception is on screen, whether the user can do something with it depends on three things: their role, what role the current workflow step expects, and whether that step is the active step right now.

The three checks

Check 1 User's role appears in the user info table
+
Check 2 Current workflow step is configured for that role
+
Check 3 That step is the active step on this exception
=
Action allowed Buttons enabled, write APIs accept

Any one of the three failing means the action is denied. The user may still see the exception (entitlement passes) and may still understand what stage it's at — they simply can't move it forward unless all three align.

Why all three together

Role alone isn't enough — a Product Control user shouldn't act on a step meant for Reg Reporting. Step configuration alone isn't enough — even the right role can't act on a step that's already passed. Active-step alone isn't enough — the system needs to know who is allowed to advance it.

Where the rules live

Step-to-role mapping is part of the workflow configuration that already exists for the exception type. The active step is on the exception row itself (current workflow_step). The user's role is on the user info table refreshed at login. Nothing new is invented — the gate is just a clean evaluation of three existing facts.

Worked example

Exception: a TB-BB exception on master book MB_4471, business area L4 FX_OPTIONS_AMER, currently on the "PC Review" step.

Front Office user (alice.gid): sees the exception (entitled via FO pathway), but the current step is configured for the Product Control role. Action: denied. She can view, comment if comments are open, but cannot advance the step.

Product Control user (paul.gid): sees the exception (entitled via L4 pathway as PC), the step is configured for PC, and it's the active step. Action: allowed. He can approve, reject, or send back.

Reg Policy user (rita.gid): sees the exception (entitled via L4 pathway as Reg Policy), but the active step is "PC Review", not her step. Action: denied — pending. When the workflow advances to "Reg Policy Review", her gate flips to allowed.

§ 07 — Migration & Decommission

How we retire the existing structure

Controlled cut-over with a parallel-verification window. The new model runs alongside the old one for long enough to prove equivalence, then the old one is removed in one cut.

Stand up the new path in shadow

Deploy the FO entitlements table, the L4 entitlements table, and the user info table. Run the transformations against production sources but route no consumer traffic to them yet.

Reconciliation harness

For every user-and-exception pair the existing structure currently authorizes, assert the new path authorizes the same. Run nightly. Target: zero discrepancies for two consecutive runs before flag-flip.

Flag-flip the read path

Switch the app to the new resolution logic via feature flag. Monitor latency, error rate, and ticket volume for 48 hours. Rollback is one flag-flip away.

Decommission the legacy

After one week of clean operation: drop the legacy entitlement tables, archive their final state, remove the dead code, retire the related batch jobs, update the runbook.

Open items for review

— IT role scoping: do Prime IT, Non-Prime IT, and Central IT CVS and AVS users see all exceptions, or are they scoped by something we haven't captured yet?

— Effective dating: are L4 assignments expected to be time-versioned (effective from / to) on day one, or is "current state with audit history" sufficient for v1?

— Stale role behavior: if the Entitlement Service is unreachable on login, is fall-back to last-known roles acceptable for a bounded window, or should the app refuse access?

— Governance Forum (RO): is read-only access enforced at API level (no write endpoints exposed), at workflow gate level (Check 2 always denies), or both?
