§ 03 — Access resolution

The rule per role, in pseudocode

if user_type == "Front Office Owner": // book-level filter via NPL FO data show exceptions where main_exception.sdsbookcode IN ( select MasterBookId from npl_fo_data where user.gid IN (FO_Owner, Delegate, Supervisor, JointSup) ) elif user_type == "Product Control": // L4-level filter via PC L4 data show exceptions where main_exception.businessarea IN ( select L4 from pc_l4_data where PC_Owner = user.gid ) elif user_type == "Governance Forum": // read-only · no filter at all show all exceptions disable all action buttons (RO) else: // Risk · Regulatory · IT central team show exceptions where main_exception.workflow_initiated = true // UI further filters the visible workflow steps // to those configured for the user's role
§ 04 — Action gating

Seeing ≠ acting

Check 1user_type present in user_info
+
Check 2step's role config includes user_type
+
Check 3step is the active step on the exception
=
Action allowedButtons enabled, write APIs accept

The view filter from §03 controls visibility; this gate controls action. They are independent. Governance Forum users always fail Check 2 by design (no step is configured for that role) — the gate is the single enforcement point for the read-only behavior.

§ 05 — Notifications

L4 governance is read on every step transition

on exception step transition (exception_id, new_step): L4 = main_exception[exception_id].businessarea discipline_columns = step_to_governance_column_map[new_step] // e.g. "PC Review" → [pc_owner_delegate, pc_supervisor] // "Reg Reporting Review" → [reg_rep_owner_delegate, reg_rep_supervisor] recipients = select discipline_columns from business_area_l4_governance where BusinessAreaL4 = L4 notify(recipients, exception_id, new_step)

Notifications dispatch outside the request thread (queue or scheduled job — match the project's existing pattern). Failure to notify never blocks the workflow transition.

What just happened

alice hits the FO branch — her query reduces to sdsbookcode IN ('MB_4471') because that's the only book where her GID is the FO_Owner. Two rows.

paul hits the PC branch — his query reduces to businessarea IN ('FX_OPTIONS_AMER'). Two rows.

rachel and ian hit the catch-all else branch — workflow_initiated = true. They miss EX-1004 because it has not been initiated yet.

gary hits the Governance Forum branch — no filter at all. He is the only user who sees EX-1004. Every action button on his screen is disabled by the gate in §04.

Action gating on the same data

alice can act on EX-1001 (active step "FO Review" matches her role).

paul can act on EX-1002 (active step "PC Review" matches his role).

rachel can act on EX-1005 (active step "Risk Review" matches her Risk role). She cannot act on EX-1003 even though she sees it, because "Reg Reporting Review" is not configured for the Risk role.

ian sees the same four exceptions as rachel but cannot act on any of them — none of the active steps in this snapshot is configured for IT central team.

gary never has Check 2 pass. Read-only across the board.
