S 02 · Entitlement fetch on login

Local DB first, then Python filewatcher, then 403

The UI calls our local entitlement service to identify the logged in user. If the local store has nothing for them, the Python filewatcher service is invoked synchronously to call the finportal entitlement API, persist any returned roles into user_info_roles, and return them. If finportal does not recognize the user, the response is 403 Forbidden.

def resolve_user_roles(brid): // Step 1. Local lookup. user = select * from user_info where brid = brid roles = select role_name from user_info_roles where brid = brid if user and roles: return {brid, is_admin: user.is_admin, roles} // Step 2. Ask the Python filewatcher service to fetch fresh. result = filewatcher_client.fetch_user_entitlements(brid) // filewatcher calls finportal, persists, and returns roles in one round trip. if result.status == "NOT_FOUND_IN_FINPORTAL": raise HTTP403("Forbidden. User not recognized in finportal") if result.status == "NO_TBBB_ROLES": // User exists but holds no TB BB role. Block at landing. return {brid, is_admin: false, roles: []} // Step 3. Return what was just persisted. return {brid, is_admin: result.is_admin, roles: result.roles}

Python filewatcher responsibilities

The filewatcher already runs as a service. It picks up two responsibilities here. First, on demand fetch when called by the UI for a user with no local roles. Second, eventual bulk processing if and when finportal can deliver a daily entitlement file or Autosys can extract one. The on demand path is the interim; the bulk path will replace per user lookups for known users once available.

def fetch_user_entitlements(brid): // Calls finportal entitlement API. resp = finportal_api.get_user(brid) if resp.status_code == 404: return {status: "NOT_FOUND_IN_FINPORTAL"} all_roles = resp.json().roles tbbb_roles = [r for r in all_roles if r.name contains "TB BB"] is_admin = any(r.name == "ADMIN-TB BB-USER-RW" for r in all_roles) if tbbb_roles: upsert_user_info(brid, is_admin) for role in tbbb_roles: insert_user_info_role(brid, role.name) return {status: "OK", brid, is_admin, roles: tbbb_roles}

The local store is the cache, not the source of truth. Roles persist between sessions but a refresh pathway should exist (manual button or scheduled refresh) so that role removals in finportal eventually reflect locally without waiting for the next miss.

S 03 · The three view filters

Each view is a separate query scoped by primary role

The dashboard loads three lists for every user. Each list runs its own query. The branches below cover all roles. Note that acted_by_log is used by Actions Required (to suppress already acted on items for an FO user) and by My History (to surface only personal actions).

Actions Required
if primary_role == "Front Office": // Pre workflow exceptions on books they own. return exceptions where sdsbookcode in user_books(brid) and workflow_initiated = false elif primary_role == "Product Control": // On their L4 and currently at PC Review. return exceptions where businessarea in user_l4s(brid) and workflow_step = "PC Review" elif primary_role in (Risk_*, Reg_Reporting, Reg_Policy, IT_Central): // Workflow initiated and active step is configured for this role. return exceptions where workflow_initiated = true and workflow_step in steps_for_role(primary_role) elif primary_role == "Governance Forum": return [] // always empty
All Exceptions
if primary_role == "Front Office": // Every exception on books they own, any stage. return exceptions where sdsbookcode in user_books(brid) elif primary_role == "Product Control": // Every exception on their L4, any stage. return exceptions where businessarea in user_l4s(brid) elif primary_role in (Risk_*, Reg_Reporting, Reg_Policy, IT_Central): // Workflow initiated, plus the role is currently active OR has been completed // previously by another user with the same role. return exceptions where workflow_initiated = true and ( workflow_step in steps_for_role(primary_role) or exists( acted_by_log[exception.id] contains an entry where role = primary_role ) ) elif primary_role == "Governance Forum": return exceptions // no filter
My History
// Identical for every primary role except Governance Forum. if primary_role == "Governance Forum": return [] return exceptions where exists( acted_by_log[exception.id] contains an entry where actor_brid = current_user.brid )

About acted_by_log. Conceptually it is a JSON array on the exception or a side table exception_action_log with rows (exception_id, actor_brid, actor_role, action, action_at). The latter is cleaner for queries and audit. Pick whichever fits the existing schema, but the All Exceptions and My History queries depend on it.

Visibility (S 03) and the action gate are independent. A PC user sees an exception at FO Review stage (visibility passes via GMIS), but Check 2 fails because the step is not configured for PC. S 05 · Notifications and SLA

Two triggers, one wide table

The business_area_l4_notification table is laid out wide. One row per L4. Two columns per role: an owner column for the BRID or DL that should receive regular notifications, and a supervisor column for SLA escalation. Layout:

Step transition (regular)

on step_transition(exception_id, new_step): L4 = main_exception[exception_id].businessarea role = role_for_step(new_step) owner_col = role + "_owner" recipients = select owner_col from business_area_l4_notification where L4 = exception.L4 notify(to = recipients, exception_id, new_step) // supervisor column NOT touched here

SLA breach (escalation)

on sla_breach_detected(exception_id): L4 = main_exception[exception_id].businessarea step = main_exception[exception_id].workflow_step role = role_for_step(step) owner_col = role + "_owner" sup_col = role + "_supervisor" row = select owner_col, sup_col from business_area_l4_notification where L4 = exception.L4 notify(to = row[owner_col], cc = row[sup_col], exception_id, step, breach=true)

SLA detection. A scheduled job compares each open exception's time at step against sla_config.sla_hours for the active step. When the threshold is crossed, the exception's sla_state flips to BREACHED and the breach trigger fires once. Subsequent runs do not re notify on the same breach.

S 06 · Admin role

Parallel role for configuration

ADMIN-TB BB-USER-RW

Admin is a parallel finportal role. A user with user_info.is_admin = true has unlocked write access to two configuration UIs and their backing tables.

Notification configuration screen writes to business_area_l4_notification. SLA configuration screen writes to sla_config. All other Admin write APIs return 403.

Admin is independent of primary role. The S 03 view resolution selects branches based on primary_role only. The is_admin flag does not appear in any view query, never enters the action gate, and grants no L4 or book level entitlement. A PC plus Admin user still sees only PC's L4 in the exception views.

Audit. Both configuration tables are @Audited with Hibernate Envers. Every Admin change writes a revision tagged with the user BRID, so any modification to recipient or threshold values is traceable.

Risk and Reg Reporting branch. rachel sees five rows in All Exceptions: EX-2003 and EX-2010 are at Credit Risk Review (active for her role), and EX-2006, EX-2007, EX-2008 are exceptions where Credit Risk completed (her role's prior involvement is in acted_by_log).
