Fix 2 — DB-Backed Async Job Queue · Weeks 1–3 · Core Architectural Fix
2
DB-Backed Async Job Queue — MADB
RECOMMENDED · Weeks 1–3
Replace the synchronous HTTP call with a persistent row in MADB. FOBO inserts and returns in <10ms. Titan workers poll for the next available job ordered by priority_order, claiming it atomically. Jobs survive node crashes via ACID guarantees and lease expiry auto-recovery.
  • Threads freed in <10ms — down from 10+ min
  • ACID crash resilience — lease expiry auto-reclaims timed-out jobs
  • Checkpoint resume — no Stage 1 full restart on failure
  • Zero new infrastructure — FOBO and Titan already use MADB
  • Regional priority column — APAC/EMEA/AMER close-order enforced
  • Full SQL observability — queue depth, SLA, backlog, region breakdown
Pattern proven in production: Solid Queue (37signals), pg-boss (Node.js), db-scheduler (Java), Oban (Elixir).

Rule: When a book from an earlier-closing region arrives late into a later-closing region's window, it jumps the queue — its deadline is sooner.   Late APAC during EMEA → pre-empts all EMEA books.   Late EMEA during AMER → pre-empts all AMER books.   AMER books have no later region to be late into.

Key InsightThe MADB job tracking table is required regardless of transport choice. DB polling builds this foundation with zero additional infrastructure. RabbitMQ can be layered on later — replacing only the transport, not the state machine.
Why Not Kafka?Kafka's max.poll.interval.ms (default 5 min) is below our 10-min job duration — consumer ejected mid-job, causing duplicate processing. Also needs a 3-broker cluster for 0.35 msg/sec. Significant complexity for no gain here.
