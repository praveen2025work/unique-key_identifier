How skills and MCP tools work together in the FOBO agent

Concept
Skill vs MCP tool — two different things
Skill = a prompt template. It tells the LLM what to do and in what order. No code runs. It lives in .claude/commands/ as a markdown file.

MCP tool = a callable Python function. It actually executes: loads data, runs logic, returns JSON. It lives in src/mcp_server.py as a @mcp.tool() decorated function.

The skill triggers LLM reasoning. The LLM then decides which MCP tools to call and in what order. Neither works without the other.
Skill .claude/commands/recon.md
Run a complete FOBO recon cycle for trade date: $ARGUMENTS Step 1: Call reconcile_all(...) Step 2: For each HIGH/MEDIUM break → call create_exception(...) Step 3: Skip AUTO_RESOLVE breaks Step 4: Call generate_report(...) Step 5: Write plain-English summary
MCP tool src/mcp_server.py
@mcp.tool() def reconcile_all( trade_date: str, book: str = "ALL" ) -> dict: """Loads FO+BO, matches, classifies all breaks.""" fo = data_layer.get_fo_trades(...) bo = data_layer.get_bo_settlements(...) results = reconciler.match_all(...) return {"summary": ..., ...}
Startup — step 1 of 3
Claude Code reads .mcp.json and starts the server
When you run claude in the project folder, Claude Code looks for .mcp.json in the working directory. It reads the server config and spawns the MCP server as a child process via stdio.
Config .mcp.json
{ "mcpServers": { "fobo": { "command": "python", "args": [ "src/mcp_server.py" ], "env": { "PYTHONPATH": "." } } } }
Process What Claude Code does
1. Read .mcp.json 2. Spawn subprocess: python src/mcp_server.py 3. Connect stdin/stdout pipes (this IS the MCP transport) 4. Send: initialize request 5. Wait for: initialize response 6. Send: tools/list 7. Store all tool schemas in memory Server is now "connected"
Startup — step 2 of 3
The tools/list handshake — what Claude Code learns
FastMCP automatically generates JSON schemas from your Python function signatures and docstrings. Claude Code receives these schemas and stores them. The LLM can now reference all 5 tools by name.
JSON-RPC tools/list request (Claude Code → server)
{ "jsonrpc": "2.0", "id": 1, "method": "tools/list" }
JSON-RPC tools/list response (server → Claude Code)
{ "result": { "tools": [ { "name": "reconcile_all", "description": "Run the complete FOBO reconciliation...", "inputSchema": { "type": "object", "properties": { "trade_date": { "type": "string" }, "book": { "type": "string", "default": "ALL" } } } }, { "name": "create_exception" }, { "name": "get_exceptions" }, { "name": "reload_rules" }, { "name": "generate_report" } ] } }
Startup — step 3 of 3
Claude Code also discovers the skills
Claude Code scans .claude/commands/ for markdown files. Each file becomes a slash command. When the user types /recon, Claude Code reads that file and expands $ARGUMENTS with whatever the user typed after the command name.
Skill discovery .claude/commands/
recon.md → /recon recon-desk.md → /recon-desk reload-rules.md → /reload-rules exceptions.md → /exceptions Each .md file: - First line = command description - $ARGUMENTS = user's input text - Rest = instructions to the LLM

Execution — step 1 of 3
LLM reads the skill and generates a tool_use block
The LLM (Claude) reads the expanded skill text as instructions and the tool schemas as its vocabulary. It generates a tool_use content block specifying which tool to call and with what arguments. Claude Code intercepts this and executes the tool via MCP.
LLM output Claude's first response
{ "content": [ { "type": "tool_use", "id": "toolu_01", "name": "reconcile_all", "input": { "trade_date": "2025-05-19", "book": "ALL" } } ], "stop_reason": "tool_use" }
MCP wire Claude Code → fobo server (stdio)
{ "jsonrpc": "2.0", "id": 2, "method": "tools/call", "params": { "name": "reconcile_all", "arguments": { "trade_date": "2025-05-19", "book": "ALL" } } }
Execution — step 2 of 3

FastMCP receives the JSON-RPC message, finds the reconcile_all function by name, validates inputs against the Pydantic schema, calls the function, and wraps the return value as a tool result.

Execution — step 3 of 3
LLM reads the result and loops — calling create_exception for each break
Claude Code sends the tool result back to the LLM as a tool_result message. The LLM reads it, follows the skill instructions, and generates the next tool_use block. This loop repeats until the skill is complete.
MCP wire Tool result → LLM
{ "role": "user", "content": [{ "type": "tool_result", "tool_use_id": "toolu_01", "content": "{ 'summary': { 'total_breaks': 5, 'high': 3, 'total_pnl_impact': 3173700 }, 'results': [ {'fo_trade_id':'TRD-002', 'break_type':'QTY_BREAK', 'severity':'HIGH', 'auto_resolvable':false}, ... ] }" }] }
LLM decides Next tool calls (follows skill)
Skill says: "For each HIGH/MEDIUM break, call create_exception" LLM sees 5 breaks. TRD-002 QTY_BREAK HIGH → tool_use TRD-003 PRICE_BREAK HIGH → tool_use TRD-005 MISSING_IN_BO HIGH → tool_use TRD-004 STATUS_MISMATCH LOW → skip (LOW severity) After all exceptions created: → tool_use: generate_report → stop_reason: end_turn → Write plain-English summary
Full picture
The complete call chain for one /recon cycle
Skills and MCP tools have completely different jobs. The skill is the brain — it tells the LLM what to orchestrate. The MCP tools are the hands — they actually execute. You can change either independently.
