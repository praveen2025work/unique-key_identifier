Custom views rendered as MCP Apps inside the AOF chat interface.

Prohibited activities -- hard-coded in governance layer
No automated regulatory submissions to PRA/FCA without human sign-off. No direct payment or fund transfer execution. No financial statement certification or sign-off. No modification of GL entries. No circumvention of Delegation of Authority limits. These blocks are enforced by the governance layer before any tool runs -- your plugin cannot override them. Design all write actions with human approval gates.
The simplest way to understand MCP servers
Think of a hospital.

AOF Generic MCP Servers are the shared hospital services available to every doctor -- the X-ray machine, the blood test lab, the medical library. Any doctor in the building can use them. They are not specific to cardiology or neurology. They do general things every department needs.

Your Plugin MCP Servers are the cardiology department's own equipment -- the ECG machine, the cardiac catheter lab, the heart-specific patient records. Only cardiologists have them. They speak the language of hearts, not general medicine.

The Agent is the doctor. It uses both -- shared services AND specialist equipment -- depending on what the patient needs. But it only accesses the cardiology equipment when it is operating in the cardiology plugin (your RBAC role).
AOF generic MCP servers -- available to all users right now

Document search and retrieval using regex and semantic matching. Extracts relevant passages, matches entities, finds related content across uploaded documents.

DuckDB SQL engine, statistics, ML calculations, charts and exports. Takes any tabular data and runs aggregations, comparisons, and visualisations.

PageIndex Generic
Semantic indexing for long documents. Creates L0/L1/L2 summaries for multi-level document Q&A.
Use: index workflow process docs, enable agent to answer questions about your procedures
Documents Generic
Creates DOCX, XLSX, HTML, PPTX files. Generates formatted reports and exports from structured data.
Use: generate a formatted variance report, create adjustment summary Excel for reviewer
Brave Search Generic
Web and news search. LLM context search and deep research on external topics.
Use: look up market commentary, check external rate data, research regulatory guidance
Reg Scanner Generic
5 jurisdictions. Regulatory change detection and impact analysis against your documents.
Use: scan for relevant regulatory changes, check if new rules affect your close process
What generic servers cannot do
They cannot connect to your internal Barclays systems. They do not know what your specific systems, books, balances, or workflow steps are. They work on data you give them -- they cannot go and fetch your live data from your internal tools. That is what your plugin MCP servers do.
Your plugin MCP servers -- what your team builds
Your System MCP Server 1 Your team builds
Connects to your primary internal system (e.g. position/book management system). Read-only. Knows your data model. Returns live business data in structured JSON the agent can reason about.
Example tools:
get_book_status(book_id, date)
get_reconciliation(book_id, date)
get_position_summary(book_id, date)
Your System MCP Server 2 Your team builds
Connects to your workflow tool and secondary data systems. Read-only. Knows your process model -- steps, statuses, balances, file arrivals.
Example tools:
get_workflow_step_status(date)
get_balance(book_id, date)
check_items_generated(book_id, date)
get_file_arrival_status(date)
What your MCP servers can do that generic ones cannot
They speak your system's language. They know your data model. They return business-meaningful data. The agent calls your tool, gets a structured result (e.g. book status: open/closed, balance: 1,250,000, reconciliation: BREAK with reason), then uses Claude Sonnet to explain it in plain English to your users. The generic Data Analysis server then runs variance calculations on that data. The Documents server generates the Excel output. Each layer does what it is best at.

The key insight
Your plugin MCP servers provide the live internal data. AOF generic servers do the analytical heavy lifting on that data. The agent orchestrates both. You only need to build the connectors to your systems -- not the analytics engine, not the document generator, not the search infrastructure. AOF provides all of that already.

What your team builds -- 5 components
1
MCP Server 1 -- Primary internal system connector
A Python FastAPI (or Node.js) service that connects to your primary internal system. Stateless, read-only by design. Exposes tools the agent can call to fetch live data. Follows the MCP protocol format that AOF expects.
# Define your tools first -- before writing code:
get_book_status(book_id, date) -- is this item open or closed?
get_reconciliation(book_id, date) -- any breaks or mismatches?
get_summary(book_id, date) -- current position/balance summary
get_breaks_detail(book_id, date) -- detail on any exceptions
2
MCP Server 2 -- Workflow and secondary system connector
Connects to your workflow tool and any secondary data systems. Same stateless, read-only pattern. Returns workflow state, step completion status, file arrival confirmation, and item generation status.
get_workflow_step_status(date) -- which steps are done/blocked?
get_balance(book_id, date) -- current balance figure
check_items_generated(book_id, date) -- are items ready?
get_file_arrival_status(date) -- have all input files arrived?
get_pending_approvals(date) -- what is waiting for human sign-off?
3
Plugin Skills -- What the agent knows how to do
Define the skills your plugin exposes to the AOF orchestration engine. Each skill is a named capability that maps to one or more MCP tool calls. Skills tell the agent what it can accomplish within your plugin.
Skills to define:
- Status Check: are all books/items open and ready?
- Balance Validation: is the balance correct vs expected?
- Reconciliation Check: any breaks between systems?
- Item Status Report: are items generated and correct?
- Summary Generation: produce the user-facing status report
- Workflow Step Monitor: which of the N steps are complete/blocked?
4
Plugin UI -- Status dashboard as MCP App
A custom view that renders inside the AOF chat as an MCP App. Shows the current status from all your checks in one screen. Includes an approval queue for any write actions that need human sign-off before executing.
Screens to build:
- Status Dashboard: overall status at a glance (all checks, any exceptions)
- Approval Queue: pending human approvals for write actions
- Step Monitor: visual view of workflow progress
5
Users & Roles -- RBAC definition
Declare which roles have access to your plugin and what each can do. Submitted to the AOF governance layer. Plugin-specific roles are scoped entirely to your plugin -- they do not affect other plugins.
Roles to define:
- Plugin Admin: configure plugin, manage user access, set thresholds
- Domain User (e.g. Producer): full read access, trigger checks, view status
- Reviewer: view status dashboard, approve or reject write actions only
What you do NOT need to build
Authentication and SSO, session management, audit logging, approval gate infrastructure, governance policy enforcement, rate limiting, the agent itself, AWS Bedrock integration, document generation, data analysis engine, web search, storage layer. All of this is provided by AOF. Your plugin registers with it and declares what it can do. AOF handles the rest.
End-to-end data flow -- from user query to response
1
User opens AOF web chat -- SSO authentication creates a unique session UUID. No separate login. Session data isolated per UUID, auto-cleared after 1 hour.
2
User types a query or uploads a file -- File uploads go to S3 and are text-extracted, then indexed via RAG Server and PageIndex for semantic search within the session.
3
Policy Compliance Check -- Governance layer runs before anything else. RBAC Plugin Access Validation confirms the user has access to your plugin. Tool Governance Check confirms the tools being called are permitted for this user and session.
4
Agent One routes the query -- Determines whether the query requires your domain plugin or generic services. Routes to your plugin's skills and MCP servers if it is a domain-specific question.
5
Your plugin MCP servers are called -- Agent calls your tools (get_book_status, get_balance, get_reconciliation, etc.). Your MCP servers query your internal systems and return structured JSON results.
6
Generic MCP servers are called as needed -- Agent calls AOF Data Analysis to run calculations on your returned data. Calls Documents server to generate an Excel or Word output. All of this happens in the same agent session automatically.
7
Write action triggers approval gate -- If the agent wants to submit or upload anything, it pauses. An approval record is created in your plugin's approval queue. The agent cannot proceed until a named reviewer approves. This gate is enforced by AOF's governance layer.
8
Output policy screening -- All outputs screened against policy constraints. Any prohibited action blocked before response is returned to the user.
9
Formatted response returned -- Status summary displayed in Plugin UI. Artifacts stored in S3 for download. Every action -- tool call, approval, token spend -- logged in the audit trail automatically.
Open question -- scheduled / programmatic invocation
The current AOF architecture is web chat initiated -- a user starts the session. There is no visible scheduled trigger in the current design. However the Claude Agent SDK on AWS Bedrock technically supports programmatic invocation. Ask the AOF platform team: "Does AOF support scheduled agent runs or programmatic session creation via API?" If yes -- your workflow check can run automatically every morning and have a status summary ready before users arrive. If not -- it is a high-value feature request to raise.
Frequently asked questions about AOF and plugin development
What exactly is AOF and how does it differ from other AI tools?
▶
AOF -- Agent One Finance -- is the firm's own AI agent platform, built on AWS Bedrock using the Claude Agent SDK. It is not a third-party tool or an external SaaS product. It runs inside the firm's infrastructure with full governance, RBAC, audit trails, and SSO already in place. Three finance domain plugins are already live and in use. New plugins follow a documented, standardised pattern that the AOF platform team has defined.
Is AOF approved for use, or does it need additional sign-off?
▶
AOF is already approved, already running, and already used by multiple finance teams across the firm. Building a plugin on AOF means building on the already-approved platform following its documented plugin pattern. The governance layer, RBAC, prohibited activities list, and audit trail are all already in place -- any new plugin inherits them automatically. No new AI tool is being introduced.
What happens if the agent makes a wrong decision or submits something incorrectly?
▶
The agent does not make final decisions on submissions or approvals. It gathers information and surfaces it. Any write action -- any upload or submission -- requires explicit human approval before it executes. The AOF governance layer enforces this. The agent's job is to do the legwork; the user makes the call and is the named approver on record. This is by design, not optional.
How is an AI agent different from the automation we already have?
▶
Existing automation runs fixed steps in a fixed order -- it is a script. An AI agent can reason. If a system is unavailable, it waits and retries. If an item is missing, it explains what is missing and why, not just a red flag. If a reconciliation breaks, it can explain the break based on what it read from the source system. Automation handles the happy path. The agent handles the exceptions -- which is where users spend most of their time.
What does the team need to build versus what AOF already provides?
▶
AOF already provides the agent engine, governance layer, RBAC, audit trail, SSO, AWS Bedrock, document generation, and data analysis tools. None of that needs to be rebuilt. What a team builds is specific to their domain: MCP servers that connect the agent to internal systems, a set of plugin skills that define what the agent can do, a status dashboard UI, and the user role definitions. The heavy infrastructure is already there.
How long does it take to build a plugin, and what does it cost?
▶
A prototype (two MCP servers and one core skill) takes roughly 3 to 4 weeks. A full plugin (all skills, Plugin UI, full user roles) takes 3 to 4 months. The recommended approach is to start with the prototype to prove value before committing to the full scope. Infrastructure cost is near zero -- the plugin runs on AOF's existing AWS Bedrock and shared infrastructure. The cost is engineering time to build the MCP connectors and skills.
Why does the domain team need to build this -- can't the platform team do it?
▶
The AOF platform is generic -- it provides the plumbing. The domain knowledge is the hard part. The domain team understands the data model of the internal systems, the sequence of checks that need to happen, and what the approval chain looks like. If the domain team does not define the plugin, either it does not get built or someone else defines it incorrectly. Domain knowledge is the differentiator.
How is governance and compliance handled?
▶
Plugins are built on the firm's own approved AI platform, not an external tool. The governance layer, RBAC, prohibited activities list, and audit trail are all already in place and every plugin inherits them. All write actions -- any submission or upload -- are gated behind a human approval step. The agent surfaces and recommends; the user decides and approves. This is how AOF is designed to work.
Can this benefit other teams beyond the one building the plugin?
▶
Each plugin is scoped to its own domain and users. However, the MCP server pattern established and documented by one team becomes a reference for any other team that wants to build a plugin on AOF. Early movers set the standard that other teams follow. That is a strong position to be in as the firm scales AOF across more domains.
Can AOF plugins run automatically without someone triggering them manually?
▶
Current AOF architecture is web chat initiated -- a user starts the session from the browser. Whether scheduled or API-triggered invocation is supported should be confirmed with the AOF platform team. The underlying Claude Agent SDK on AWS Bedrock supports it technically. If it is not yet available, it is a high-value feature request -- a morning automated status run before users arrive would significantly increase the value of any plugin.
What are the first steps to get started?
▶
Two things are needed. First -- an introduction to the AOF platform team to get the Plugin Development Guide and register intent to build. Second -- time to build a prototype: two MCP servers and one core skill. Three to four weeks is enough to demonstrate the pattern working end-to-end and get stakeholder sign-off before committing to the full plugin scope.
