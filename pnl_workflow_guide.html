Named PNL & Book Mapping

A Named PNL is the fundamental unit of P&L ownership. Product Controllers (PCs) own one or more Named PNLs. Each is a logical container for a set of Master Books tracked in RAMP and processed daily by FAS against the MOTIF database.

ConceptDefinitionSystem
Named PNLLogical grouping of Master Books for P&L reporting. Created and owned by users. e.g. "DA Funding and Discarded"RAMP
Master BookA financial ledger in Chorus (front-office system). Multiple books map to one Named PNL. Feeds arrive from front-office via TDB.Chorus / TDB
RAMPUpstream configuration system. Users define Named PNL ↔ Master Book mapping here. Fully editable in production with immediate effect.RAMP
FASThe service/workflow layer. No separate database. All data reads/writes go through MOTIF DB. Calls RAMP at SOD to fetch book mapping, caches in FAS cache.FAS / Workflow
MOTIF DBCentral back-office database. Source of truth for balances, book status, journal entries, adjustments. All FAS SQL queries target this.MOTIF
GMISGroup Management System. The downstream submission target. Validates FTD = Σ P-Lex categories on upload. Also provides GMIS Review reports for user validation pre-submission.GMIS
Example: Named PNL "DA Funding and Discarded" has 122 Master Books. FAS calls RAMP at SOD, fetches and caches the 122-book list. All subsequent daily processing runs against those 122 books until next SOD refresh.
2

Daily Workflow Stages

Every business day, the following stages execute sequentially for each Named PNL. FAS orchestrates the full pipeline; approximately 628 distinct extract artefacts are produced across all Named PNLs daily.

Stage 1 — SOD Role (Start of Day)
FAS calls RAMP to fetch the current Master Book list for the Named PNL. Result is cached in FAS. SOD balance snapshots (FTD=0, MTD and YTD from yesterday's finalised values) are created for each book.
Stage 2 — Book Open for Correction
FAS polls the MOTIF book status table every 5 minutes. Books must receive all feeds (Trades, Cash Flows, Valuations, Instruments, Starting Balance) from front-office systems via TDB before moving to "Open for Correction." Once all books open, stage completes and Rec Factory trigger file is created.
Stages 3–4 — Mandatory Functionalities
~15 mandatory features that are enabled for all Named PNLs regardless of asset class or trading desk. These are system-level prerequisites that must pass before any PC activity is permitted for the day.
Stages 5–10+ — Extract Generation
FAS fires SQL queries to MOTIF DB to generate the 8 core extract reports: FX Rates, FX Exposure, Adjustments, Book Completeness, Trial Balance, P-Lex, Consolidated Validation, GMIS Review. Reports are written to the user path for retrieval.
PC Review & Edit
Product Controllers review extracts. They post adjustments (permanent or reversing) and/or reclassify P-Lex explain categories via Finance Portal or Workflow upload templates (Excel). Both actions write directly to MOTIF.
Tollgate 1 — Lock & Submit to GMIS
PC formalises P&L and attests at Tollgate 1. FAS calls GMIS/MOTIF to lock all books (status → Assigned Locked for Edit). Final Trial Balance + P-Lex + GMIS Review are submitted. Tollgate 2 rejection returns books to "Undersigned" status.

4

Core Data Extracts

Eight major data artefacts are produced daily for each Named PNL. All data is sourced from MOTIF DB via FAS SQL queries. Extracts are deposited into the user path and optionally consumed by the GMIS submission flow.

Extract 1
FX Rates
Daily currency conversion rates sourced from the MOTIF FX rate table. Foundation data for FX Exposure calculation. One rate per currency pair per business day.
Source: MOTIF FX rates table
Extract 2
FX Exposure
Today's P&L balance converted to the reporting currency (e.g. 100 GBP → USD) using current FX rates. Currency config derives from Chorus accounting calendar per book — not a FAS-configured parameter.
Source: MOTIF balance + FX rates
Extract 3
Adjustments
All active adjustments for today's business date across all mapped books. Separates permanent adjustments (no expiry) from reversing adjustments (auto-reverses after N days). Used as PC reference for reviewing outstanding entries.
Source: MOTIF adjustments table
Extract 4
Book Completeness
Validates all Chorus books mapped to this Named PNL are present in the workflow. Orphaned books (in Chorus but not tracked) are flagged. A completeness check only — no financial values.
Source: RAMP mapping vs Chorus registry
Extract 5
Trial Balance
Core financial report. Daily FTD / MTD / YTD balance per Master Book. Sourced from MOTIF's Opening Balance and Journal Line Item tables. Drillable by GL, SL Account, Cost Centre, or Chorus aggregation level.
Source: MOTIF JLI + Opening Balance tables
Extract 6
P-Lex (P&L Explains)
Explains every dollar of Trial Balance FTD across ~200 explain categories (Interest, Commission, Risk Amount, Brokerage, etc.). GMIS enforces: Σ P-Lex categories must equal FTD. Any residual is flagged as "Unexplained P&L."
Source: MOTIF P-Lex assignment tables
Extract 7
Consolidated Validation
A combined view of FX Exposure + P-Lex + Adjustments in one report. Also exposes the two edit vectors: manual adjustment posting and explain category reclassification. Provides the "full picture" before GMIS submission.
Source: Composite of Extracts 2, 3, 6
Extract 8
GMIS Review
Submission-ready view combining Trial Balance + P-Lex. This is the file uploaded to GMIS. GMIS validates FTD = Σ explains before accepting. Residual unexplained P&L amounts surface here and must be addressed before submission.
Source: Extracts 5 + 6; GMIS validation
Extra
Flash vs Formal
Trader-published flash P&L estimate vs PC-formalised FTD. Deviation >250 GBP requires PC commentary per attestation policy. Data available in workflow output files for analytical comparison.
Source: Trader flash + Trial Balance FTD
5

PC User Edit Actions

Product Controllers perform two types of edits during the daily workflow. Both write immediately to MOTIF DB and affect all subsequent extract outputs.

ActionPurposeTypesEntry Point
Manual Adjustment Post a delta (+/−) to a book's FTD balance when feed data doesn't match expected business value Permanent — retained indefinitely

Reversing — auto-reverses after N days		 Finance Portal (Adjustments screen) or Workflow upload template (Excel)
Explain Reclassification Move balance between explain categories. E.g. $20 from "Interest Income" → "Trade Amendment." P-Lex total unchanged. Always immediate; affects GMIS Review P-Lex balance Finance Portal (Reclass screen) or Workflow upload template (Excel)
Re-run Gap: There is currently no automated re-run of extracts after a user posts an adjustment or reclass. Users must manually regenerate the report, or use the GMIS Review screen to see the updated figure. Some PCs skip the re-run and validate directly from GMIS Review before Tollgate 1. This is a known functional gap flagged for future remediation.
Attestation Threshold: Single adjustment value >250 GBP requires an attestation entry in RAMP per policy. The Adjustments extract and Consolidated Validation report are the primary audit trail for this.

Analytics & Reporting Use Cases

Five high-value reporting use cases derivable from existing workflow data. All data is already in MOTIF DB — this is a pure visualisation and aggregation layer requiring no new data capture.

📊
Report 1 — Book OFC Timing Analysis (Feed Delay Detection)
Source: Book OFC stage completion timestamps · MOTIF status table

Track the time each Named PNL achieves "Open for Correction" status each day. Deviations from the expected window signal front-office feed delays or MOTIF processing issues.

✓ On time⚠ Minor delay✗ Feed delay
  • Record OFC timestamp per Named PNL per business day into a historical table
  • Flag days where OFC time exceeds a configurable threshold (e.g. >2 hours past expected window)
  • Compare OFC times week-on-week and month-on-month to identify systemic patterns or degradation
  • Map delays to specific regions — APAC opens earliest, EMEA next, Americas last; regional delays are distinguishable
  • Distinguish holiday closures (holiday=Yes, expected) from genuine feed failures (holiday=No, books still closed late in day)
  • Generate weekly trend report: "Books for PNL X opened 47 min later this week vs prior week average"
📉
Report 2 — Adjustment Volume & Threshold Monitoring
Source: Adjustments extract (Extract 3) · MOTIF adjustments table

Monitor adjustment count and total value per Named PNL over time. High volumes indicate poor feed quality or recurring business issues. Adjustments >250 GBP individually trigger RAMP attestation policy.

  • Count of adjustments (permanent + reversing) per Named PNL per business day
  • Total GBP value of adjustments — flag when single adjustment exceeds 250 GBP attestation threshold
  • Week-on-week trend: is adjustment count increasing (feed quality degradation) or decreasing (upstream fix applied)?
  • Identify PNLs that chronically post high volumes — candidates for front-office data quality review
  • Separate reversing vs permanent breakdown to identify structural vs one-off corrections
⚡
Report 3 — Flash vs Formal Deviation Tracker
Source: Flash data (trader-published) vs finalised FTD from Trial Balance

Traders publish a flash P&L estimate each day. The PC's finalised FTD should be within a configured tolerance of this flash value. Deviations beyond the threshold require documented commentary.

  • Compare flash FTD vs finalised FTD per Named PNL per day; calculate absolute and percentage deviation
  • Flag Named PNLs where deviation exceeds configurable threshold (e.g. >250 GBP)
  • Surface commentary captured in workflow against each deviation for audit trail
  • Trend: is deviation pattern between trader and PC improving or worsening over time?
  • Identify books where flash is consistently inaccurate — suggests trader estimation methodology issues
❓
Report 4 — Unexplained P&L Monitor
Source: GMIS Review extract (Extract 8) · FTD vs Σ P-Lex categories

Every dollar of FTD must be attributed to a P-Lex explain category. Residual unexplained amounts are a GMIS submission blocker and indicate incomplete P-Lex attribution.

  • Unexplained Amount = FTD − Σ(all explain categories) per Named PNL per day
  • Flag PNLs where unexplained amount exceeds a configurable per-PNL threshold (Finance to define, e.g. >$50)
  • Trend: is unexplained balance shrinking during the day (user reclassing) or static (user not acting)?
  • Identify PNLs that chronically carry unexplained balances — signals P-Lex category mapping gaps in MOTIF config
  • Intraday notification: alert PC when unexplained amount exceeds threshold so they can reclass before EOD
🚦
Report 5 — GMIS Submission Readiness Dashboard
Source: All stages — OFC status, adjustments, P-Lex balance check, Tollgate 1 status

Single-pane view for PC, BTP, or manager showing which Named PNLs are submission-ready vs blocked and why. Aggregatable across multiple PNLs for team-level visibility.

  • Per Named PNL: Book OFC status · Adjustment count today · P-Lex balance check (FTD = Σ explains?) · Tollgate 1 status
  • RAG (Red/Amber/Green) indicator per PNL — Green = ready for submission, Amber = issues present, Red = blocked
  • Drill-down: which specific books within a PNL are still "Open for Input" and blocking OFC completion?
  • Compare today's readiness status against same day prior week / prior month for anomaly detection
  • Aggregate across multiple Named PNLs for BTP / Product Control Manager-level oversight view

MCP Servers Required

Each MCP server is a FastAPI adapter exposing typed tools to the AOF agent. These wrap existing FAS/MOTIF capabilities — no new backend logic is needed, only an MCP protocol adapter layer.

MCP Server 1
MOTIF DB MCP

FastAPI adapter exposing read-only MOTIF Oracle DB queries via JDBC. Used by all three agents. Wraps the existing SQL queries FAS already executes.

get_book_status(pnl_id, date) get_trial_balance(pnl_id, date) get_plex(pnl_id, date) get_adjustments(pnl_id, date) get_fx_rates(date) get_ofc_timestamp(book_id, date) get_journal_line_items(book_id, date)
MCP Server 2
RAMP API MCP

Wraps RAMP REST API to expose Named PNL configuration and book mapping data. Read-only for agents; write operations remain in RAMP UI.

get_named_pnls(user_id) get_book_mapping(pnl_id) get_holiday_calendar(region, date) get_attestation_threshold(pnl_id)
MCP Server 3
GMIS API MCP

Wraps GMIS REST API to expose submission status, validation results, and review data. Enables agent to check GMIS readiness and surface validation failures to PC.

get_gms_review(pnl_id, date) get_submission_status(pnl_id, date) validate_plex_balance(pnl_id, date) get_unexplained_pnl(pnl_id, date)
MCP Server 4
Notification MCP

Enables agents to fire alerts to Teams channels and email. Supports threshold-based alerting (book delay, adjustment breach, unexplained P&L) and scheduled summary reports.

send_teams_alert(channel, message) send_email_summary(recipients, report) create_scheduled_job(trigger, agent_id)
MCP Server 5
FAS Service MCP

Wraps FAS's existing service API to trigger extract regeneration (addressing the re-run gap) and query workflow stage status. Provides the agentic re-run capability currently missing from the workflow.

get_stage_status(pnl_id, date) trigger_extract_rerun(pnl_id, extract_type) get_workflow_history(pnl_id, days)
MCP Server 6
Analytics Store MCP

A lightweight Oracle 19c historical analytics schema (separate schema in MOTIF DB or dedicated schema). Persists daily snapshots of OFC times, adjustment counts, deviations — enabling the trend analytics reports in Section 7.

store_daily_snapshot(pnl_id, date, metrics) get_ofc_trend(pnl_id, weeks) get_adjustment_trend(pnl_id, weeks) get_flash_deviation_trend(pnl_id)

MCP Server Implementation Pattern (FastAPI)

Python · FastAPI MCP Adapter — MOTIF DB MCP
# MOTIF_mcp_server.py — FastAPI MCP adapter for MOTIF Oracle DB
from fastapi import FastAPI
from mcp.server.fastapi import MCPRouter
from pydantic import BaseModel
import cx_Oracle, os

app = FastAPI(title="MOTIF DB MCP Server")
router = MCPRouter()

pool = cx_Oracle.SessionPool(
    user=os.getenv("MOTIVE_USER"),
    password=os.getenv("MOTIVE_PASS"),
    dsn=os.getenv("MOTIVE_DSN"),
    min=2, max=10, increment=1
)

@router.tool(description="Get current status of all books for a Named PNL on a given date")
async def get_book_status(pnl_id: str, business_date: str) -> dict:
    query = """
        SELECT mb.book_id, mb.book_name, bs.current_status,
               bs.ofc_timestamp, hol.is_holiday, hol.region
        FROM   master_book mb
        JOIN   book_status bs  ON mb.book_id = bs.book_id
                              AND bs.business_date = TO_DATE(:bdate, 'YYYY-MM-DD')
        JOIN   pnl_book_map pm ON mb.book_id = pm.book_id AND pm.pnl_id = :pnl
        LEFT JOIN holiday_calendar hol ON hol.business_date = bs.business_date
                              AND hol.region = mb.region
        ORDER  BY mb.book_name
    """
    with pool.acquire() as conn:
        with conn.cursor() as cur:
            cur.execute(query, bdate=business_date, pnl=pnl_id)
            cols = [c[0].lower() for c in cur.description]
            return {"books": [dict(zip(cols, row)) for row in cur.fetchall()]}

@router.tool(description="Get Trial Balance FTD/MTD/YTD for a Named PNL")
async def get_trial_balance(pnl_id: str, business_date: str,
                             drill_level: str = "book") -> dict:
    query = """
        SELECT ob.book_id, ob.account_level, ob.gl_account,
               ob.ftd_balance, ob.mtd_balance, ob.ytd_balance
        FROM   opening_balance ob
        JOIN   pnl_book_map pm ON ob.book_id = pm.book_id AND pm.pnl_id = :pnl
        WHERE  ob.business_date = TO_DATE(:bdate, 'YYYY-MM-DD')
        AND    ob.drill_level   = :level
    """
    with pool.acquire() as conn:
        with conn.cursor() as cur:
            cur.execute(query, pnl=pnl_id, bdate=business_date, level=drill_level)
            cols = [c[0].lower() for c in cur.description]
            return {"trial_balance": [dict(zip(cols, row)) for row in cur.fetchall()]}

app.include_router(router, prefix="/mcp")

Skills, Plugins & AOF Integration

AOF Plugin: P&L Controller Plugin

The P&L Controller Plugin is the primary AOF plugin that registers the agent's tools, system prompt, and trigger configuration into the AOF platform on AWS Bedrock.

AOF Plugin
P&L Controller Plugin
Registered in AOF with the system prompt defining agent persona, tool bindings (MCP servers), memory context (Named PNL list, user role), and scheduled triggers (SOD, EOD, mid-day check-in). This is the entry point for all P&L agent interactions.
AOF Plugin
Book Monitoring Plugin
Separate plugin with a scheduled trigger: runs every 5 minutes during business hours. Checks OFC status for all active Named PNLs. Fires Teams alerts when delay threshold is exceeded. Low-latency, narrow scope — purposely separate from P&L Controller for isolation.
Skill (Claude CLI / AOF)
PNL Workflow Knowledge Skill
SKILL.md-style prompt file injected as context when agent answers workflow questions. Contains the book status lifecycle, extract definitions, P-Lex rules, GMIS validation rules, and attestation thresholds. Enables the agent to interpret data correctly without hallucination.
Skill (Claude CLI / AOF)
Analytics Reporting Skill
Defines the 5 report templates (OFC timing, adjustment monitor, flash-formal, unexplained P&L, readiness dashboard). Includes threshold configs, RAG logic, and output format. Used by Reporting Agent to structure outputs consistently for Finance Portal and email delivery.
