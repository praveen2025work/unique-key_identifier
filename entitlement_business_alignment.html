01

Why we are changing this

Today's entitlement structure conflates two different kinds of access into one shape. The result is brittle joins, manual reconciliation, and confusing accountability when an exception lands in the wrong queue. This refactor separates them cleanly.

  • Two access paths, one source of truth FO ownership flows from NPL Ramp data automatically. Governance-role access (PC, MR, WCR, CCR, IRB, Reg Policy, Reg Reporting) is admin-managed at Business Area L4. The new model honors how the business actually thinks about ownership.
  • Less duplication in the data NPL Owner + NPL Delegate become a single FO Owner column. SupervisorGID + JointSupGID become a single Supervisor column. Five source columns collapse into three — easier to read, easier to audit.
  • Fewer manual touches for governance teams L4-level role assignments (Owner/Delegate, Supervisor) for each governance discipline are maintained centrally — by admin UI or file upload — instead of being scattered across spreadsheets.
  • Clean retirement of the existing structure The current entitlement table set will be decommissioned in a controlled cut-over. No parallel-run drift, no dual sources of truth.
Right person → right exception → right time. That's all an entitlement system needs to do, and that's what this delivers.
02

How it will work

Two distinct pathways feed one resolved view of "who can see this exception." Each pathway has a clear source, a clear owner, and a clear join.

Figure 1 — Two pathways converging on the existing main exception table. No change is required to that table.

Pathway A

FO Owner — driven by NPL data

Source: NPL Ramp feed · Refresh: per current cadence

  1. NPL Owner and NPL Delegate are merged into a single FO_Owner column.
  2. supervisorGID and jointSupGID are merged into a single Supervisor column.
  3. Access is granted on every exception where the user appears in either column for the matching MasterBookId.
Pathway B

Governance Roles — driven by L4

Source: admin UI or file upload · Maintained by governance team

  1. For each Business Area L4, admin captures Owner-or-Delegate + Supervisor for seven disciplines.
  2. Each role becomes its own row in the resolved entitlements table.
  3. Access is granted on every exception where businessarea = BusinessAreaL4 for that user's role.

04

Who owns what

Clear ownership is the difference between an entitlement system that ages well and one that breaks every time the business changes shape. Here's how we'd like to split it.

Business / Governance teams

  • Confirm the canonical role names per discipline.
  • Provide the initial Business Area L4 governance file (Owner/Delegate + Supervisor per role).
  • Designate admin users authorized to maintain L4 assignments.
  • Sign off on UAT scenarios (FO and L4 paths).
  • Agree the cut-over window for the existing structure retirement.

Finance Technology

  • Build the two transformation pathways and the resolved entitlements model.
  • Build the admin UI and bulk-upload validator for L4 governance data.
  • Decommission the existing entitlement structure once parallel verification passes.
  • Provide an audit trail for every entitlement change (who, what, when).
  • Deliver runbook and support documentation.
Decision needed

Role naming

Confirm the seven discipline names before schema is locked. Renaming after build is cheap; renaming after rollout is not.

Decision needed

L4 admin owners

Identify a primary and a backup admin per region or function to maintain Business Area L4 assignments.

Decision needed

Cut-over window

A 1-week parallel verification window followed by a controlled retirement of the existing structure.
